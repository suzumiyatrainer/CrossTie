# RTMおよびNGTLib関連 バグ修正仕様書

## 1. 概要
RealTrainMod (RTM) および前提Modである NGTLib において、特定の条件下で発生する描画や処理の不具合（バグ）を修正するためのパッチ群です。
バグが元々発生していた技術的背景、具体的な修正アプローチ、そして修正結果について記載しています。

---

## 2. エンティティ・システム関連のバグ修正

### 2.1 クライアント側列車移動の平滑化および補間バグの自己相殺修正

- **対象ファイル**: `EntityTrainClientSmoothingMixin.java`
- **問題の背景**:
  マルチプレイサーバーにおいて、他のプレイヤーが運転している列車や自動運転中の列車は、サーバーから送られてくる位置更新パケットの間隔（通常20tickに数回）の都合上、ガクガクとコマ送りのように動いて見えていました。
  これに対してクライアント側で滑らかに移動を補間（Lerp / 線形補間）する処理を入れようと試みたものの、補間処理が正常に働かず、列車がカクついたままでした。
- **バグの根本原因**:
  位置補間処理の中で、電車の現在位置を徐々に目標座標に近づけるために `setPositionAndRotation` メソッドを呼び出しますが、このメソッドを呼んだ瞬間に「サーバーから目標値が送られてきた時」に動く目標値記録用のフック（`@Inject(at=RETURN)`）が再帰的に動いてしまっていました。
  結果として、**「補間中の中間座標（まだ目標に達していない位置）」が「次の最終目標値」として即座に上書きされてしまう現象**が発生し、補間処理が自己相殺（自己破壊）していました。
- **修正内容**:
  Mixin内部に補間適用中を示すフラグ（`crosstie$isApplyingSmooth`）を新設しました。
  補間処理として `setPositionAndRotation` を呼び出す間はこのフラグを `true` にし、その間に発生した目標値記録処理（`recordTarget`）を完全にスキップして再帰ループを遮断しました。
- **結果**:
  自己相殺バグが解消され、クライアント側での線形補間（Lerp）が正しく働くようになり、マルチプレイ時の列車の移動や旋回が非常に滑らかに描画されるようになります。

### 2.2 サウンドAPI競合によるエラーとメモリリークの修正

- **対象ファイル**: `MixinSoundAPIEntityTrainBase.java`
- **問題の背景**:
  列車の走行音、ブレーキ音、警笛（汽笛）などが正しく再生されなかったり、長時間プレイしているとゲームの動作が著しく重くなり、最終的にメモリ不足（Out Of Memory）でクラッシュすることがありました。
- **バグの根本原因**:
  RTMは列車の音響処理にSoundAPIを使用していますが、列車のエンティティがデスポーン（消滅）した際や、チャンクがアンロードされた際に、作成された音響インスタンスの破棄命令が適切にSoundAPI側に伝わっていませんでした。これにより、サーバー/クライアント上に解放されないオーディオソースが残り続け、メモリリークと再生チャンネルの枯渇を招いていました。
- **修正内容**:
  列車エンティティの廃棄処理（`setDead` や `onChunkUnload` 等）にフックし、関連付けられているSoundAPIの全再生チャンネルとオーディオリソースを即座かつ安全にクローズ・解放する処理を追加しました。
- **結果**:
  電車の音が出なくなるバグが解決し、長時間のプレイでもオーディオ関連のメモリリークが発生しなくなります。

## 3. 描画・レンダリング関連のバグ修正

### 3.1 カリング（描画省略判定）の誤作動によるオブジェクト消失バグ修正

- **対象ファイル**: `TileEntitySignalNoCullingMixin.java`, `TileEntityCrossingGateNoCullingMixin.java`, `RenderEntityInstalledObjectCullingMixin.java`
- **問題の背景**:
  RTMで設置した信号機、踏切の遮断機、および設置物（InstalledObject）に近づいたり、プレイヤーが視点を左右に振ったりした際、オブジェクトが画面の端（あるいは特定の角度）に入った瞬間に不自然にパッと消えてしまう（カリングされる）バグがありました。
- **バグの根本原因**:
  MinecraftおよびRTMは描画負荷を抑えるために視界外のオブジェクトを描画しない「カリング」を行います。しかし、信号機の高いアームや踏切の長い遮断棒、複雑なInstalledObjectの3Dモデルは、システム上のバウンディングボックス（描画可否を判定するための見えない箱）のサイズ計算が実際のモデルサイズと一致しておらず、極めて小さく設定されていました。そのため、本体が画面内にあっても「描画境界の外にある」と誤判定されていました。
- **修正内容**:
  - 対象のTileEntityおよびエンティティのバウンディングボックスの拡張・再計算を適切に行うようにフックを記述しました。
  - 特に誤判定が発生しやすい信号機・踏切に関しては、カリング判定を強制的に無効化（常に描画対象とする）し、描画パイプラインへ確実にモデルデータを送るように介入しました。
- **結果**:
  視点を動かした際に信号機や踏切が不自然に消えたり点滅したりする描画崩れが完全に解消されます。

### 3.2 車両モデル描画時の行列（Matrix）バグ修正

- **対象ファイル**: `RtmPartsMatrixPushMixin.java`, `BasicVehiclePartsRendererMixin.java`
- **問題の背景**:
  一部の車両追加パックや、ドアの開閉・パンタグラフの昇降といった可動パーツを持つモデルを描画する際、パーツの表示位置が大幅にずれたり、ゲームが突然クラッシュ（OpenGLの「Stack Overflow / Underflow」エラー）したりする現象が発生していました。
- **バグの根本原因**:
  RTMの車両パーツレンダラーにおいて、3D空間の位置や回転を設定するためのOpenGL行列スタック処理（`glPushMatrix()` と `glPopMatrix()`）の数が一致していませんでした。ある描画パスで行列をプッシュしたものの、条件分岐によってポップされないまま処理が終了する、あるいはその逆が発生し、OpenGLの行列状態が崩壊していました。
- **修正内容**:
  車両パーツの描画（Render）ライフサイクルを Mixin で追跡し、パーツ描画の開始時と終了時において、行列スタックの整合性が確実に保たれる（必ずプッシュとポップが対になる）ように補正コードを注入しました。
- **結果**:
  可動パーツや複雑なサブパーツを多数持つ車両パックでも、パーツ位置が狂うことなく、またOpenGLエラーによるクラッシュを引き起こさずに安全に描画できるようになります。

### 3.3 1.12.2規格マーカーにおける描画近代化MOD・シェーダー環境互換パッチ（Angelica & OptiFine）

**詳しくはこちら** → [マーカー描画不良問題の修正](/doc/各修正の概要/RTM-NGTLib/マーカー描画不良問題の修正.md)

- **対象ファイル**: `GLHelperSelectBypassMixin.java`, `RenderMarkerBlock1122MathPickingMixin.java`, `RenderMarkerBlockBaseMixin.java`, `InternalGUIMathPickingMixin.java`, `MarkerRenderState.java`
- **問題の背景**:
  近代化・高速化MODである **Angelica（GLSMによる描画近代化）** や、**OptiFine + シェーダーパック（Shaders）** 有効環境下において、1.12.2マニフェストマーカーのアンカー線（調整用の矢印）や距離の十字マーク、立方体グリッド（GRID）が完全に消滅して見えなくなるバグが発生していました。また、文字・GUIが画面を覆うほど超巨大化して表示が崩れるバグや、クリック判定（ピッキング）が一切機能しなくなるバグが併発していました。
- **バグの根本原因**:
  1. **判定消失・巨大化**: マウス判定に使用される OpenGL レガシー機能の `GL_SELECT`（選択モード）が、GLSMやシェーダーの描画最適化（FBO）と致命的に競合し、ヒット判定が常に `0` になる上、プロジェクション行列（`GL_PROJECTION`）の復帰に失敗して状態を破壊し、文字等の超巨大化を引き起こしていました。
  2. **線・マークの消滅**: テクスチャを無効化した状態での描画（`glDisable(GL_TEXTURE_2D)`）を行うと、シェーダーやGLSMによって描画情報（アンカー線やグリッド、距離十字など）が「不要なジオメトリ」としてFBOの書き込みから完全に破棄（discard）されてしまっていました。
  3. **GUIの透明度崩れ・ボタンの境界消滅**: 背景と文字で描画パス（テクスチャ有効無効、および使用するテッセレータ）が乖離したことでシェーダー上の前後関係が反転し、さらにボタン描画の内部で `glDisable` が走り以降のボタンが消滅していました。
- **修正内容**:
  1. **GL_SELECTの完全無効化と数学的ピッキングへの移行**:
     * `GLHelperSelectBypassMixin` にて、問題を引き起こす `GL_SELECT` のピッキング処理自体をバイパス（No-op化）して行列破壊を根底から防止。
     * 代わりに、`gluProject` を用いて3D空間のアンカー線やGUIボタンの座標を2Dスクリーン座標に投影し、画面中央（照準）との距離・交差から数学的判定を行うロジックに置き換え、操作性を完全復旧。
  2. **動的1x1白色テクスチャによる G-Buffer 破棄の回避**:
     * 状態管理クラス `MarkerRenderState` を新設し、1x1ピクセルのダミー白色テクスチャを動的に一度だけ生成。
     * アンカー線（`RenderMarkerBlock1122MathPickingMixin`）、距離マークとグリッド（`RenderMarkerBlockBaseMixin`）、GUI背景と各ボタン（`InternalGUIMathPickingMixin`）の描画時に、テクスチャを無効化せず、代わりにこの白色テクスチャを一時的にバインドして Tessellator 経由のパスで描画。
- **結果**:
  Angelica環境およびOptiFine Shaders環境のいずれにおいても、アンカー線やグリッド、距離の十字マークが本来のカラーで正しく描画され、GUIの透明度や文字サイズも崩れずに操作可能な状態で正常動作するようになります。

### 3.4 OptiFine+シェーダー環境における黒の電線描画不良の修正

**詳しくはこちら** → [黒の電線描画不良問題の修正](/doc/各修正の概要/RTM-NGTLib/黒の電線描画不良問題の修正.md)

- **対象ファイル**: `NGTTessellatorMixin.java`
- **問題の背景**:
  OptiFineとシェーダー（影MOD）を導入した環境において、`basicwirerenderer`（`RenderBasicWire.js`）で描画される架線（電線）などのオブジェクトが描画されず、完全に透明になって消えてしまう（ポストのみが表示される）バグが発生していました。
- **バグの根本原因**:
  1. `RenderBasicWire.js` では単色描画のため事前に `GL_TEXTURE_2D` が無効化されていました。
  2. `NGTTessellatorMixin` 側には「シェーダー有効時はジオメトリ消失を防ぐため、強制的に `GL_TEXTURE_2D` を有効化しダミーUV(0, 0)を付与する」というワークアラウンドがありましたが、この時に無地の白テクスチャを明示的にバインドしていませんでした。
  3. 結果として、直前にバインドされていたテクスチャ（ブロックアトラスなど）の (0, 0) がサンプリングされ、そこが透明ピクセルだった場合にアルファ値が0になり、描画全体が透明になって消滅していました。
  4. また、ワイヤー描画では法線（Normal）が指定されておらず、シェーダーパックによっては陰影計算に失敗して描画が破棄される問題もありました。
- **修正内容**:
  1. **白ピクセルの強制バインド**: `GL_TEXTURE_2D` を強制有効化する際、1x1サイズの「完全な白色（不透明）」テクスチャを動的に生成・バインドするようにし、設定カラー（黒色）が正確に乗るように修正しました。
  2. **状態復元とデフォルト法線**: 描画終了後に元のテクスチャバインド状態へ復元する処理を追加。さらに、法線情報を持たない描画に対しては、シェーダー環境下でダミーの法線（0.0, 1.0, 0.0）を自動付与して計算エラーを防ぎました。
- **結果**:
  OptiFine+Shader環境において、架線（ワイヤー）が消失せず本来の指定色で正常に描画されるようになります。
