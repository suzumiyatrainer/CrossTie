# Angelica および GTNHLib 関連の互換性修正仕様書

## 1. 概要
Minecraft 1.7.10 環境を大幅に近代化・最適化するGT New Horizonsプロジェクト発のコアMod群（**Angelica** および **GTNHLib**）を、RTMやその他のModと併用した際に発生する、表示の崩れ・シェーダーパック（影Mod）の不動作・クラッシュ等の互換性問題を修正するパッチ群です。

---

## 2. Angelica (Iris / Embeddium / Celeritas) 互換性修正

Angelicaは、描画パイプラインを近代的な OpenGL Core Profile ベースへと置き換えるため、従来の固定機能パイプライン（1.7.10バニラや旧Modが多用する描画手法）と深刻な競合を引き起こします。

### 2.1 ディスプレイリスト（DisplayList）競合によるJVMクラッシュの修正
- **対象ファイル**: `AngelicaRenderGlobalDisplayListCrashMixin.java`
- **問題の背景**: 
  RTMはレールなどの重いモデルを描画するために OpenGL の DisplayList を生成・管理しています。しかし、Angelicaがグラフィックメモリやレンダリングコンテキストを再構築するタイミング（描画設定の変更やディメンション移動など）で、RTM側が保持している古いDisplayList IDが不正なポインタとなってしまい、グラフィックドライバがアクセス違反を起こしてゲームが即座にクラッシュ（JVM強制終了）していました。
- **修正内容**: 
  AngelicaのレンダーマネージャとRTMのDisplayList生成・破棄のライフサイクルを同期させ、不要になったりコンテキストが失われたリストを安全に破棄・バイパスする安全網を構築しました。
- **結果**: 
  Angelica併用時に、設定変更やディメンション切り替え等でゲームが突然デスクトップに強制終了する問題が完全に解決します。

### 2.2 カスタムモデルスクリプト内の生OpenGL命令の動的リダイレクト
- **対象ファイル**: `AngelicaScriptTransformCacheMixin.java`, `ModelPackManagerScriptRedirectMixin.java`, `ScriptGlRedirector.java`
- **問題の背景**: 
  RTMのカスタムモデル（JavaScript制御）では、車体やギミックの描画に `GL11.glScalef()` などの生のLWJGL OpenGLメソッドが頻繁に直接記述されています。しかし、Angelica環境下ではすべての描画ステートが `GLStateManager` によってキャッシュ・管理されているため、スクリプトから生のOpenGL命令が割り込むと、描画状態の同期が破壊され、モデルが著しく引き伸ばされたり、全く表示されなくなったりする不具合が生じます。
- **修正内容**: 
  JavaScriptのソースコードが読み込まれたタイミング（`ModelPackManager.getScript` および `AngelicaCompat.transformScript`）で、CrossTieの `ScriptGlRedirector` が割り込みます。
  スクリプト内の `Packages.org.lwjgl.opengl.GL11.glScalef` といった記述を正規表現で解析し、Angelicaの `GLStateManager.scale` などに動的に書き換えます。この変換結果は最大512件までLRUキャッシュに保持され、変換負荷はO(1)に最適化されています。
- **結果**: 
  Angelica環境下（および影Mod有効環境下）であっても、JavaScriptで制御される複雑なカスタム車両や踏切などのモデルが、位置やスケールを崩すことなく美しくレンダリングされます。

### 2.3 光源更新時におけるレールの明るさ同期不具合の修正
- **対象ファイル**: `AngelicaRebuildSyncRTMRailMixin.java`
- **問題の背景**: 
  レール（LargeRail）の側や上に松明などの光源を置いたり撤去したりした際、バニラ環境であればレールの明るさも即座に変わりますが、Angelica環境下ではレールの部分だけが古い暗さ（あるいは明るさ）のままになり、不自然な影が残るバグがありました。
- **技術的背景**: 
  Angelicaの高速描画エンジンは、ブロックの更新が起きるとチャンク全体のメッシュを即座に再構築（`scheduleRebuildForBlockArea`）しますが、TileEntityの描画（レールのTESR）に対しては更新の通知を行いません。結果、レールのDisplayListが古いライト情報のまま維持されてしまっていました。
- **修正内容**: 
  Angelicaのメッシュ再構築処理の末尾にフックし、更新が発生した座標範囲内に存在する RTMのレール（`TileEntityLargeRailCore`）を高速走査して、描画更新フラグ（`shouldRerenderRail = true`）を強制的にセットする処理を記述しました。
- **結果**: 
  光源の設置・破壊に伴う明るさの変化が、レールの描画に対してもタイムラグなく即座に反映されるようになります。

### 2.4  シェーダー有効時におけるバニラ雲の二重描画バグ修正
- **対象ファイル**: `AngelicaCloudRenderingFixMixin.java`
- **問題の背景**: 
  Angelicaに内蔵されているIris（シェーダーローダー）でシェーダーパック（影Mod）を有効化した際、シェーダー側の雲とバニラの矩形雲が両方同時に描画され、空の見た目が極めて不自然になる、あるいは激しく点滅するバグがありました。
- **修正内容**: 
  `RenderGlobal.renderClouds`（およびFancy/Fast雲描画）の開始時に、Iris의 シェーダーパイプラインがアクティブ（有効）であるかを検出し、有効な場合はバニラの雲描画処理を完全にキャンセル（早期リターン）するように介入しました。
- **結果**: 
  シェーダーパックが適用されている間はバニラの不要な雲描画が完全に非表示になり、シェーダー側の空だけが美しく描画されます。

### 2.5 描画距離の境界における水面表示の乱れ修正
- **対象ファイル**: `AngelicaRenderDistanceWaterFixMixin.java`
- **問題の背景**: 
  Iris（シェーダー）有効時、描画距離（Render Distance）の最果てにあるチャンクにおいて、水面のレンダリングが乱れ、水面が透明になって底が透けたり、チラつきが発生するバグがありました。
- **修正内容**: 
  ワールド描画処理（`renderWorld`）の開始時に OpenGL の背面カリング（`GL_CULL_FACE`）を `GLStateManager` を通して強制的に有効化し、描画完了後に元の状態に戻すことで、長距離描画時の水面の描画ステートの不一致を定式化しました。
- **結果**: 
  遠方にある川や海などの水面が、描画距離の境界線上でも破綻することなく正常にレンダリングされます。

### 2.6 大文字・小文字ケース不一致によるシェーダー適用バグの修正
- **対象ファイル**: `AngelicaBlockMaterialMappingCaseFixMixin.java`, `AngelicaItemMaterialHelperCaseFixMixin.java`
- **問題の背景**: 
  Minecraft 1.7.10 の一部のMod（例：「竹Mod」の `BambooMod`など）は、Modの登録IDのドメイン名に大文字を維持した状態で登録されています。しかし、Angelica（Iris）のシェーダー割り当て機構（`BlockMaterialMapping` / `ItemMaterialHelper`）は標準でIDをすべて小文字（例：`bamboomod`）に強制変換した上でレジストリを探索するため、ブロックの検出に失敗し、シェーダーパックで設定された特殊効果（水面反射や草木の揺れなど）が適用されないバグがありました。
- **修正内容**: 
  Angelicaがブロック/アイテムIDを解決するメソッド（`resolveBlockOrNull` 等）に介入し、レジストリ内の登録名に対して大文字・小文字を区別しない（`equalsIgnoreCase`）比較を行うように修正しました。
- **結果**: 
  竹Modの温泉水や、その他のドメイン名に大文字が含まれるModのオブジェクトに対しても、シェーダーパックの特殊レンダリング（波、反射、揺れ等）が正しく適用されるようになります。

### 2.7 シェーダーIDマップの解析順序およびOptiFine互換化
- **対象ファイル**: `IdMapOrderFixMixin.java`
- **問題の背景**: 
  シェーダーパックに付属する `block.properties` や `item.properties`（どのブロック/アイテムにどのシェーダーIDを割り当てるかのマッピング）の解析結果が、OptiFine環境下とAngelica環境下で異なり、一部のブロックに正しいテクスチャや光沢が適用されない問題がありました。
- **修正内容**: 
  - ブロックマップの解析結果を格納するデータ構造を順序が保証されない `Int2ObjectOpenHashMap` から、挿入順が保持される `Int2ObjectLinkedOpenHashMap` に変更し、プロパティファイルに書かれた優先順位通りにIDが適用されるようにしました。
  - アイテムマップの格納時において、重複したキーがある場合は「最初に見つかった割り当て」を優先する（`putIfAbsent` に変更する）ことで、OptiFineと全く同一の挙動になるようにしました。
- **結果**: 
  OptiFine環境向けに設計された複雑なリアル系シェーダーパックであっても、Angelica環境下で完璧な互換性を持ってブロックやアイテムが描き分けられます。

## 3. GTNHLib 互換性修正

GTNHLibはパフォーマンス向上用の内部APIを提供しますが、スレッド設計の違いなどから競合を引き起こすケースがあります。

### 3.1 オブジェクトプール（ObjectPooler）のスレッドセーフ化によるクラッシュ修正
- **対象ファイル**: `ObjectPoolerThreadSafeMixin.java`
- **問題の背景**: 
  GTNHLibは、描画やオブジェクト生成にかかるJavaのメモリ確保（GC）を削減するために、使い回し可能な配列やクラスインスタンスを蓄積しておく `ObjectPooler` を持っています。しかし、このプールはシングルスレッドでの使用を前提として設計されており、非同期チャンクロードやマルチスレッド描画、別スレッドからの書き込みが発生した際に競合（`ConcurrentModificationException` 等）を起こし、サーバーやクライアントがクラッシュする重大な不具合がありました。
- **修正内容**: 
  `ObjectPooler` 内のオブジェクト蓄積・引き出し処理をスレッドセーフな実装（排他制御）に変更し、複数の非同期スレッドが同時にプールへアクセスしてもデータ構造が破壊されないように安全化しました。
- **結果**: 
  GTNHLib導入環境下でのマルチスレッド処理実行時の安定性が飛躍的に向上し、原因不明の突発的なクラッシュが防止されます。

### 3.2 ブロックペイン（BlockPane）等のアイコン欠損によるクラッシュ防止
- **対象ファイル**: `MixinBlockPaneFix.java`, `MixinBlockPaneIconFallback.java`, `MixinBlockIconFallback.java`
- **問題の背景**: 
  GTNHLibが薄型ブロック（板ガラスや鉄格子など）の描画最適化を適用する際、何らかの理由でModブロック of アイコン（テクスチャデータ）が一時的に取得できなかった場合に、ヌルポインタ参照（NullPointerException）を起こしてゲームが強制終了する不具合がありました。
- **修正内容**: 
  アイコンデータの取得処理において、返されたオブジェクトがヌルである場合はバニラのデフォルトアイコン（Missing Textureアイコン）を安全な代替テクスチャとして割り当てるフォールバック機構を挟みました。
- **結果**: 
  テクスチャの読み込み不良が発生した場合でも、ゲームがクラッシュすることなく安全に代替画像（紫と黒のチェッカー柄）で描画が続行されます。
