# RTMおよびNGTLib関連 パフォーマンス最適化仕様書

## 1. 概要
RealTrainMod (RTM) および前提Modである NGTLib におけるパフォーマンス低下の原因を根本から解決し、処理を最適化するためのパッチ群です。
大量の列車やレール、架線が存在する環境でのFPS（描画パフォーマンス）およびMSPT（サーバー処理負荷）を向上させます。

---

## 2. 列車（EntityTrainBase）および同期処理の最適化

### 2.1 列車処理全体の最適化
- **対象ファイル**: `EntityTrainBaseOptimizationMixin.java`
- **問題の背景**: 
  RTMの列車エンティティは毎ティックごとに物理演算や各種ステータスのチェックを繰り返します。特にマルチプレイ等で多数の列車が設置されている場合、これらが累積してサーバーの処理時間（MSPT）を大幅に増加させ、サーバーラグを引き起こしていました。
- **最適化内容**: 
  列車のアップデート処理（`onUpdate` 内）において、不要な毎ティックの再計算（位置情報の変更がない場合の冗長な座標計算や無駄なオブジェクト作成）を間引き、ロジックを最適化しました。
- **結果**: 
  サーバー側での列車処理負荷が軽減され、多数の車両がワールドに存在する場合でもTPSの低下を防ぎます。

### 2.2 速度同期処理のパケット間引き
- **対象ファイル**: `EntityTrainBaseSpeedSyncMixin.java`
- **問題の背景**: 
  列車の正確な速度と位置を同期するため、サーバーとクライアント間で頻繁にパケットが送信されていました。しかし、速度に変化がない場合や、微小な変化しかない場合でも同期処理が走り続け、ネットワーク帯域とパケット処理スレッドを圧迫していました。
- **最適化内容**: 
  列車の速度変化がない場合や、閾値未満の極めて微細な変化しかない状況において、不要な速度同期パケットの送信をスキップするよう制御を加えました。
- **結果**: 
  ネットワークトラフィックが削減され、回線の細いマルチプレイ環境やサーバーのパケット送受信負荷が緩和されます。

### 2.3 車両トラッキング（VehicleTracker）の間引き
- **対象ファイル**: `VehicleTrackerThrottleMixin.java`
- **問題の背景**: 
  バニラのEntityTrackerと同様に、RTMは独自の `VehicleTrackerEntry` を用いて、列車の詳細な移動情報を全クライアントに定期送信（`sendLocationToAllClients`）します。デフォルトの更新頻度では、クライアント側の描画に影響を及ぼさないレベルの極めて頻繁な位置パケットが送信されており、ネットワークトラフィックの主要な原因となっていました。
- **最適化内容**: 
  `VehicleTrackerEntry` のパケット送信頻度を `THROTTLED_UPDATE_FREQUENCY`（3 tickに1回）に強制的に間引く処理（スロットリング）を実装しました。
- **結果**: 
  ネットワーク負荷が大幅に軽減され、特にマルチプレイ時に列車が動くことで発生する回線ラグ（帯域の圧迫）が顕著に改善されます。

### 2.4 在線検知センサー（EntityTrainDetector）の更新間隔制御
- **対象ファイル**: `EntityTrainDetectorThrottleMixin.java`
- **問題の背景**:
  RTMの在線検知センサー（`EntityTrainDetector`）は毎Tick、レール下最大8マスのTileEntityをスキャンして列車の在線を判定します。センサーが多数設置された複雑な信号システムでは、この走査処理がサーバーのMSPT（Tick処理時間）を圧迫していました。
- **最適化内容**:
  スロットリングカウンター（`crosstie$tickCount`）を導入し、設定値（デフォルト: 4tick間隔）の倍数のTickのみ在線スキャンを実行するように制御します。最高速度320km/h（≒88.9m/s）の列車でも4tick間の移動距離は約1.4m（レール1ピッチ未満）であるため、検知精度への実質的な影響はありません。

> [!NOTE]
> 保安システムに直結する `EntityATC` はスロットリング対象外です。この最適化は在線検知センサーのみに適用されます。スロットリングは `CrossTieConfig` の `detectorThrottlingEnabled` / `detectorThrottleInterval` で制御可能です。

- **結果**:
  信号システムが複雑な大規模鉄道ネットワーク環境でのサーバー処理負荷（MSPT）が軽減されます。

## 3. レール描画の最適化（LargeRail / Tessellate）

### 3.1 レール描画リスト（DisplayList）の再コンパイル最適化
- **対象ファイル**: `RenderLargeRailOptimizationMixin.java`, `RailPartsRendererOptimizationMixin.java`
- **問題の背景**: 
  RTMのレール（LargeRail）は、描画負荷軽減のためにOpenGLの「DisplayList（ディスプレイリスト）」を使用しています。しかし、レールのTESR（TileEntitySpecialRenderer）において、描画リストの更新フラグの管理が甘く、レールに変化がない場合でも不必要にリストの再構築（再コンパイル）が走ってしまい、FPSの大幅な低下を招いていました。
- **最適化内容**: 
  描画リストの有効フラグと更新契機を精密に制御し、レール自体やライトレベルに変更がない限り、絶対にDisplayListの再コンパイルを発生させないよう最適化しました。
- **結果**: 
  複雑な配線や多数のレールが密集する駅・車両基地周辺での描画負荷が劇的に低下し、FPSが向上します。

### 3.2 頂点計算（Tessellate）のキャッシュ化
- **対象ファイル**: `RailTessellateOptimizationMixin.java`
- **問題の背景**: 
  レール形状に沿った3Dモデルの頂点を生成する際、非常に重い三角関数演算を伴う頂点計算処理（Tessellate）が繰り返されます。この計算はレールを配置した時点で確定するものであるにもかかわらず、描画スレッド等で重複して呼び出され、CPUのボトルネックとなっていました。
- **最適化内容**: 
  レールモデルの頂点計算結果を一時的なメモリキャッシュに保持し、頂点情報が必要になった際は再計算せずにキャッシュから直接データを引き出して描画バッファへ書き込むようにしました。
- **結果**: 
  レールのレンダリングにかかるCPU時間が削減され、描画パフォーマンスが向上します。

## 4. 架線・電柱の処理および電力網の最適化

### 4.1 電線・架線モデルの接続キャッシュ
- **対象ファイル**: `RenderElectricalWiringConnectionCacheMixin.java`, `BlockLinePoleConnectionCacheMixin.java`
- **問題の背景**: 
  RTMの電線（ElectricalWiring）および架線柱（LinePole）は、周囲のどのノードと接続されているかを検知して3Dラインや碍子モデルを描画します。この「接続状況の探索と3Dモデルの幾何学計算」は毎フレーム行われており、電線が張り巡らされたワールドでは深刻なFPS低下を引き起こしていました。
- **最適化内容**: 
  電線の接続構造およびモデル形状の計算結果をキャッシュ（メモリ保持）し、ブロックの設置・破壊・状態変化といった明示的な更新がない限り、キャッシュを再利用して描画を完了させるように改修しました。
- **結果**: 
  電線を多数設置した大都市や鉄道網沿いでのFPSの低下が大幅に緩和されます。

### 4.2 お飾り架線（Decorative Wiring）の処理スキップ
- **対象ファイル**: `ElectricalWiringDecorativeMixin.java`
- **問題の背景**: 
  RTMの電線システムは、信号や電力を伝えるための伝達ネットワーク（シグナル伝達処理 `propagateSignal()`）を内部に持っています。しかし、ユーザーが「見た目（装飾）」のためだけに電線を引いた場合であっても、電線の更新時にネットワーク全体の在線や接続を追跡する高負荷な再帰処理（BFS/幅優先探索）が走ってしまい、サーバーのMSPT（Tick処理時間）を圧迫していました。
- **最適化内容**: 
  シグナルの発信起点から接続を追跡し、電力網（DIRECT接続＝電源や信号機器などへの直接接続）が1件も存在しないサブネットを「お飾り（装飾専用）架線」と自動判別するアルゴリズムを導入しました。お飾りと判定された架線は、重い `propagateSignal()` 処理から完全に除外（早期リターン）されます。
  判定結果は `WeakHashMap`（GCフレンドリーな自動解放マップ）で保持され、接続が切れたり変更されたりした瞬間にのみキャッシュを自動クリアして安全性を担保します。
- **結果**: 
  電飾や架線柱の配線など、動作しない「見た目だけ」の電線がワールドに大量に引かれていても、サーバー負荷（MSPT）がほとんど上昇しなくなります。

## 5. レールマップ計算のキャッシュ化

- **対象ファイル**: `RailMapCustomCacheMixin.java`
- **問題の背景**: 
  列車の台車位置や車体傾斜を計算するため、RTMはレールの敷設形状データを持つ `RailMapCustom` に頻繁にアクセスし、レールの全長・任意のインデックスにおける3D座標・高さ・回転（Yaw/Pitch/Roll）を取得します。この計算は極めてホットな（高頻度に実行される）パスであり、内部で非常に高コストな算術関数が動くため、列車の走行中に顕著なCPU負荷を引き起こしていました。
- **最適化内容**: 
  `RailMapCustom` の各種 getter（`getLength`, `getRailPos`, `getRailHeight`, `getRailYaw`, `getRailPitch`, `getRailRoll`）に包括的なキャッシュを適用しました。
  キャッシュの有効期限はグローバルな「Tickエポック（`crosstie$tickEpoch`）」によって管理されており、毎Tick自動的にクリアされます。これにより、同一Tick内での重複する高コスト計算は、2回目以降O(1)のキャッシュ参照へと置き換わります。
- **結果**: 
  多数の列車が走行している時のCPU負荷（特にメインスレッド）が劇的に削減され、移動時のカクつき（マイクロスタッター）が解消されます。

## 6. JavaScript実行環境およびGUIの最適化

### 6.1 JavaScript実行エンジン（ScriptUtil）のフォールバック・キャスト回避
- **対象ファイル**: `ScriptUtilInvocableCacheMixin.java` (NGTLib `ScriptUtil`)
- **問題の背景**: 
  RTMのカスタムモデル等の多くは、JavaScriptによる挙動記述（Script）が使用されており、レンダリング時に高頻度でスクリプト内の関数（`doScriptFunction`）が呼び出されます。この際、毎回 `Invocable` インターフェースへのキャストが発生し、実行のボトルネックとなっていました。また、特定のJava 8ランタイム環境下でJavaScriptエンジン（Nashorn等）が標準搭載されていない場合に、スクリプトのロード自体が失敗する問題もありました。
- **最適化内容**: 
  - ホットな描画パス上での `Invocable` キャストの無駄なオーバーヘッドを回避する実装へ変更しました。
  - Nashornが提供されていないか、クラスローダーの構成が複雑な環境でも動作するよう、代替のJSエンジンファクトリやクラスローダーを順次スキャンする堅牢なフォールバック機構を導入しました。
- **結果**: 
  スクリプト制御モデルが多数描画される際のFPS性能が向上すると同時に、Javaの実行環境（JRE/JDK）の差異によるスクリプト読み込みエラーが防止されます。

### 6.2 GUIモデル選択でのテクスチャページング
- **対象ファイル**: `GuiSelectTexturePagingMixin.java`
- **問題の背景**: 
  追加の車両パックやモデルパックを大量に導入している場合、RTMのモデル選択GUIなどを開く際に数百〜数千のテクスチャを一括で読み込もうとするため、ゲームが一時的に完全にフリーズ（応答なし）していました。
- **最適化内容**: 
  GUI上に現在表示されているページに必要なテクスチャのみをオンデマンドで読み込む「ページング（Paging）」機構を導入し、不要なテクスチャ読み込みを遅延・間引き処理するように変更しました。
- **結果**: 
  大量のモデルパックを導入している環境でも、モデル選択GUIがフリーズすることなく瞬時に開くようになり、動作が著しく安定します。
